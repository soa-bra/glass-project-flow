# Planning smart type persistence map

هذا PR تصحيحي بعد دمج PR #328.

## المشكلة
PR #328 جعل عناصر smart factories تستخدم UUID قابل للحفظ، لكن بعض أنواع العناصر الذكية قد لا تطابق قيم `element_type` المدعومة في قاعدة البيانات، مثل:

- `kanban`
- `thinking_board`
- `voting`
- `smart_text_doc`

إذا وصلت هذه القيم مباشرة إلى `planning_elements.element_type` فقد يفشل الحفظ.

## التصحيح المطلوب
- حفظ النوع المدعوم في قاعدة البيانات مثل `smart_doc` أو `entity_card`.
- حفظ النوع الأصلي في `metadata.smartType` أو حقل مكافئ.
- عند تحميل العنصر من قاعدة البيانات، يعاد بناء نوع الواجهة الأصلي من `smartType`.

## حالة الخطة
- منفذ: PR #328 أضاف UUID للعناصر الذكية.
- هذا PR: يفتح مسار تصحيح mapping لأنواع العناصر الذكية.
- المتبقي بعده: إغلاق UUID لباقي مسارات الإنشاء: النص، القلم، اللصق، الفريم، ثم guard التحويل.
